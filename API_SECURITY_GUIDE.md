# API Security Implementation Guide

## 🔐 Multi-Layer Security Architecture

Your insurance loyalty system now implements **TWO layers of security** for maximum protection:

### Layer 1: API Token (Application-Level Security)
**Purpose**: Ensures only your authorized frontend application can access the API

- **Header**: `X-API-Token`
- **Token**: `axis_loyalty_secure_token_2024`
- **Scope**: Required for ALL API requests
- **Protection**: Prevents unauthorized applications from accessing your API

### Layer 2: User Authentication Token (User-Level Security)
**Purpose**: Identifies and authenticates individual users

- **Header**: `Authorization: Bearer {token}`
- **Token**: Generated by Laravel Sanctum after successful login
- **Scope**: Required for protected endpoints only
- **Protection**: Ensures users can only access their own data

## 🛡️ How It Works

### Public Endpoints (API Token Only)
These endpoints require the API token but not user authentication:

```
POST /api/auth/send-otp
POST /api/auth/verify-otp
POST /api/auth/verify-policy
```

**Request Example:**
```bash
curl -X POST http://localhost:8000/api/auth/send-otp \
  -H "X-API-Token: axis_loyalty_secure_token_2024" \
  -H "Content-Type: application/json" \
  -d '{"mobile_number": "1234567890"}'
```

### Protected Endpoints (API Token + User Token)
These endpoints require BOTH tokens:

```
GET /api/auth/me
POST /api/auth/logout
```

**Request Example:**
```bash
curl -X GET http://localhost:8000/api/auth/me \
  -H "X-API-Token: axis_loyalty_secure_token_2024" \
  -H "Authorization: Bearer 1|abc123xyz..." \
  -H "Content-Type: application/json"
```

## 📝 Implementation Details

### Backend (Laravel)

**1. API Token Middleware** (`app/Http/Middleware/ValidateApiToken.php`)
- Validates `X-API-Token` header on every request
- Returns 401 if token is missing or invalid
- Applied to all API routes

**2. User Authentication** (Laravel Sanctum)
- Generates unique tokens for each user session
- Tokens stored in `personal_access_tokens` table
- Automatic validation via `auth:sanctum` middleware

**3. Route Protection** (`routes/api.php`)
```php
Route::middleware('validate.api.token')->group(function () {
    // Public endpoints (API token only)
    Route::post('/auth/send-otp', ...);
    
    Route::middleware('auth:sanctum')->group(function () {
        // Protected endpoints (API token + user token)
        Route::get('/auth/me', ...);
    });
});
```

### Frontend (React)

**1. API Utility** (`src/lib/api.ts`)
- Automatically adds `X-API-Token` to all requests
- Automatically adds `Authorization` header if user is logged in
- Centralized API configuration

**2. Usage in Components**
```typescript
import { apiCall } from "@/lib/api";

// Automatically includes API token
const response = await apiCall("/auth/send-otp", {
  method: "POST",
  body: JSON.stringify({ mobile_number: "1234567890" })
});

// Automatically includes both API token and user token
const response = await authenticatedApiCall("/auth/me", {
  method: "GET"
});
```

## 🚀 Production Security Checklist

### 1. Generate Strong API Token
```bash
# Generate a random 64-character token
openssl rand -hex 32
```

Update both:
- Backend: `.env` → `API_TOKEN=<generated_token>`
- Frontend: `src/lib/api.ts` → `export const API_TOKEN = "<generated_token>"`

### 2. Environment Variables
Move API token to environment variables in frontend:

```typescript
// vite.config.ts
export default defineConfig({
  define: {
    'import.meta.env.VITE_API_TOKEN': JSON.stringify(process.env.VITE_API_TOKEN)
  }
});

// src/lib/api.ts
export const API_TOKEN = import.meta.env.VITE_API_TOKEN;
```

### 3. HTTPS Only
- Enable HTTPS on production server
- Update API_URL to use `https://`
- Set secure cookie flags

### 4. CORS Configuration
```php
// config/cors.php
'allowed_origins' => [
    'https://yourdomain.com'
],
'allowed_headers' => [
    'X-API-Token',
    'Authorization',
    'Content-Type'
],
```

### 5. Rate Limiting
Add rate limiting to prevent brute force attacks:

```php
// routes/api.php
Route::middleware(['validate.api.token', 'throttle:10,1'])->group(function () {
    Route::post('/auth/send-otp', ...); // Max 10 requests per minute
});
```

### 6. Token Expiration
Configure Sanctum token expiration:

```php
// config/sanctum.php
'expiration' => 60, // Minutes (null = never expires)
```

## 🧪 Testing Security

### Test API Token Protection
```bash
# Without API token (should fail)
curl -X POST http://localhost:8000/api/auth/send-otp \
  -H "Content-Type: application/json" \
  -d '{"mobile_number": "1234567890"}'

# Expected: 401 Unauthorized

# With API token (should succeed)

  -H "X-API-Token: axis_loyalty_secure_token_2024" \
  -H "Content-Type: application/json" \

```

## 📊 Security Benefits

| Attack Type | Protection |
|-------------|------------|
| Unauthorized API Access | ✅ API Token blocks unknown applications |
| User Impersonation | ✅ User token validates identity |
| Session Hijacking | ✅ Tokens can be revoked |
| Brute Force | ✅ Rate limiting (when configured) |
| CSRF | ✅ Token-based auth (no cookies) |
| XSS | ⚠️ Store tokens securely, use httpOnly if possible |

## 🔄 Token Lifecycle

### User Authentication Flow
1. User enters mobile number
2. OTP sent (API token validated)
3. User enters OTP (API token validated)
4. **User token generated** and returned
5. Frontend stores user token in localStorage
6. All subsequent requests include both tokens
7. User logs out → token revoked

### Token Storage
- **API Token**: Hardcoded in application (frontend & backend)
- **User Token**: Stored in localStorage after login
- **Token Revocation**: Automatic on logout

## 🆘 Troubleshooting

### "Invalid or missing API token"
- Check `X-API-Token` header is being sent
- Verify token matches in both frontend and backend
- Ensure middleware is registered

### "Unauthenticated"
- Check `Authorization` header format: `Bearer {token}`
- Verify user is logged in and token is stored
- Check token hasn't expired

### CORS Errors
- Add `X-API-Token` to allowed headers in CORS config
- Ensure frontend origin is whitelisted
